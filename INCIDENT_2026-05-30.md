# Инцидент: релиз 1.13.1 зарегистрирован, но обновление не скачивалось

**Дата:** 2026-05-30  
**Длительность:** ~2 часа  
**Серьёзность:** High — update flow сломан, клиенты видят обновление, но не могут его установить  
**Сервер:** 130.49.151.52 (`api.achiemvemer.ru`)

## Симптомы

Во время выкладки релиза `1.13.1` были зафиксированы две отдельные проблемы:

1. Локальная Android release-сборка падала на этапе Gradle:

```text
Execution failed for task ':app:mergeReleaseNativeLibs'.
Configuration `:app:releaseRuntimeClasspath` contains AndroidX dependencies,
but the `android.useAndroidX` property is not enabled
```

2. После регистрации релиза через `/api/v1/update/releases` клиент получал update metadata,
   но скачивание файла заканчивалось ошибкой:

```json
{"message":"Route GET:/releases/collab_notes_1.13.1_android.apk not found","error":"Not Found","statusCode":404}
```

Во время исправления production-конфига также наблюдался кратковременный `502 Bad Gateway` от nginx
в момент перезапуска backend-контейнера.

## Причина

### 1. Android release build

В [collab_notes_app/android/gradle.properties](collab_notes_app/android/gradle.properties) не был включён AndroidX,
хотя зависимости приложения уже тянули AndroidX-библиотеки.

Итог: `flutter build apk --release` не мог завершиться успешно.

### 2. Release download в production

Проблема состояла из двух частей:

- release metadata была зарегистрирована в production API из локальной машины;
- сам APK/EXE при этом копировался только в локальный [backend/releases](backend/releases), а не в серверный volume `/app/releases`;
- на сервере при этом ещё работал старый backend, который на `/health` отдавал `version: 1.1.0` и не обслуживал актуальный static route `/releases/`.

Итог:

- update API уже сообщал клиенту о новой версии `1.13.1`;
- фактический файл на production backend отсутствовал;
- старый backend-контейнер не соответствовал текущему коду update-модуля.

## Исправление

### 1. Починка Android build

В [collab_notes_app/android/gradle.properties](collab_notes_app/android/gradle.properties) добавлено:

```properties
android.useAndroidX=true
```

После этого `publish_release.ps1` смог собрать `app-release.apk`.

### 2. Починка production download flow

Были выполнены следующие шаги:

1. На production-сервер синхронизирован актуальный backend-код:
   - `backend/src`
   - `backend/prisma`
   - `backend/package.json`
   - `backend/package-lock.json`
   - `backend/tsconfig.json`
   - `backend/Dockerfile`
   - `docker-compose.yml`
   - `nginx/app.conf`

2. В серверном `.env` обновлена версия:

```env
APP_VERSION=1.13.1
```

3. Пересобран и перезапущен production backend:

```bash
cd /opt/collab
docker compose up -d --build backend nginx
```

4. Release-артефакты загружены на сервер и скопированы в контейнер backend:

```bash
docker compose cp /tmp/collab_notes_1.13.1_android.apk backend:/app/releases/collab_notes_1.13.1_android.apk
docker compose cp /tmp/collab_notes_1.13.1_windows.exe backend:/app/releases/collab_notes_1.13.1_windows.exe
```

5. Выполнена проверка download URL:

```bash
curl -I https://api.achiemvemer.ru/releases/collab_notes_1.13.1_android.apk
curl -I https://api.achiemvemer.ru/releases/collab_notes_1.13.1_windows.exe
```

Результат: оба URL начали отдавать `HTTP/2 200`.

## Итог

Исправлены и подтверждены:

- Android release `1.13.1` собран и зарегистрирован;
- Windows release `1.13.1` собран и зарегистрирован;
- production backend обновлён до актуального кода;
- файлы релиза присутствуют в `/app/releases` внутри backend-контейнера;
- `GET /api/v1/update` возвращает корректные download URL;
- публичные ссылки на APK/EXE доступны извне.

## Рекомендации

### 1. Не регистрировать релиз до загрузки файла на production

Сначала артефакт должен оказаться в production volume `/app/releases`, и только потом — запись в `app_releases`.

### 2. Синхронизировать backend перед update-релизами

Если release storage и update API обслуживаются backend-контейнером, выкладка клиента должна включать deploy backend-кода, даже если менялся только мобильный билд.

### 3. Держать `APP_VERSION` сервера в актуальном состоянии

Проверка `/health` сразу показывает, соответствует ли production реальному релизному состоянию.

### 4. Автоматизировать production publish

Нужен единый сценарий, который делает всё атомарно:

1. build artifacts;
2. upload artifacts на сервер;
3. deploy backend/nginx;
4. copy artifacts в `/app/releases`;
5. register release в API;
6. smoke-check `GET /update` и `curl -I /releases/...`.

### 5. Не полагаться на локальный `backend/releases` как на production storage

Локальный publish создаёт только локальные файлы. Для production нужен отдельный шаг доставки артефактов на VPS.
