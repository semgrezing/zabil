**Серьёзность:** Critical — приложение полностью неработоспособно

## Симптомы

Все экраны Flutter-приложения показывали ошибку:

```
DioException [bad response]: status code 500
```

Затронутые эндпоинты:
- `GET /api/v1/groups` — загрузка групп
- `GET /api/v1/notes` — загрузка заметок
- `GET /api/v1/chats/personal` — личные чаты
- `GET /api/v1/chats/personal/:userId/messages` — сообщения

## Причина

**Рассинхронизация Prisma-схемы и PostgreSQL.**

БД изначально была создана через `prisma db push` с ранней версией схемы. Впоследствии в `schema.prisma` добавлялись новые поля, но миграции для них не создавались. При сборке нового Docker-образа Prisma Client был сгенерирован с актуальной схемой, а база осталась со старой структурой.

## Исправление

Создана миграция `20260529010000_sync_schema`, которая:
1. Добавляет все недостающие колонки (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`)
2. Создаёт отсутствующие таблицы (`CREATE TABLE IF NOT EXISTS`)
3. Создаёт индексы и внешние ключи

Файл: `backend/prisma/migrations/20260529010000_sync_schema/migration.sql`

## Рекомендации

### 1. Всегда использовать `prisma migrate dev` вместо `prisma db push`

```bash
# Правильно — создаёт файл миграции, который потом применится на проде
npx prisma migrate dev --name add_some_field

# Неправильно — изменяет БД напрямую, без записи в миграции
npx prisma db push
```

`prisma db push` удобен для прототипирования, но не создаёт файлов миграций. На проде `prisma migrate deploy` применяет только записанные миграции — если их нет, изменения не попадут в production-БД.

### 2. Проверять состояние миграций перед деплоем

```bash
# Локально — проверить что нет drift между схемой и БД
npx prisma migrate diff --from-schema-datamodel prisma/schema.prisma --to-url "$DATABASE_URL"

# На сервере — проверить какие миграции применены
docker exec collab-postgres-1 psql -U $USER -d $DB -c "SELECT migration_name, finished_at FROM _prisma_migrations ORDER BY finished_at;"
```

### 3. Добавить CI-проверку

В процесс деплоя добавить шаг, который проверяет что все миграции синхронизированы:

```bash
npx prisma migrate status
```

Если есть pending-миграции или drift — деплой должен упасть.

### 4. Не удалять старые файлы миграций

Директория `prisma/migrations/` — это полная история схемы БД. Удаление файлов приводит к рассинхронизации с `_prisma_migrations` таблицей на проде.

### 5. Добавить логирование ошибок Prisma в error handler

Текущий error handler скрывает детали 500-ошибок от клиента (правильно), но стоит убедиться что серверные логи всегда включены в production:

```typescript
// config/env.ts — убедиться что логи Prisma включены
log: ['error', 'warn'],
```
